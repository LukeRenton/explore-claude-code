# CLAUDE.md

프로젝트에 대한 지속적인 컨텍스트를 Claude에게 제공하는 마크다운 파일입니다. 매 세션마다 자동으로 로드되며, 수동으로 프롬프트할 필요가 없습니다.

## 빠른 시작

1. Claude Code에서 `/init`을 실행합니다. 코드베이스를 분석하고 기본 `CLAUDE.md`를 생성합니다
2. 생성된 내용을 검토하고 다듬습니다. Claude가 스스로 추론할 수 없는 부분을 추가합니다
3. 버전 관리에 커밋하여 팀 전체가 동일한 컨텍스트를 공유하도록 합니다
4. 시간이 지남에 따라 반복 개선합니다. 불편한 점을 발견할 때마다 지침을 추가합니다

## 무엇을 넣어야 할까

| 섹션 | 목적 | 예시 |
|---|---|---|
| 프로젝트 개요 | 코드베이스에서 Claude의 방향 설정 | 기술 스택, 아키텍처, 주요 디렉토리 |
| 빌드 및 테스트 명령어 | Claude가 워크플로우를 실행할 수 있도록 함 | `npm test`, `cargo build`, `make deploy` |
| 코드 컨벤션 | 팀의 스타일 적용 | 네이밍, 임포트, 에러 처리 패턴 |
| 일반적인 워크플로우 | 반복 작업 방지 | 브랜치 네이밍, PR 프로세스, 배포 단계 |
| 도구 및 MCP 메모 | Claude를 환경에 연결 | 커스텀 스크립트, [MCP](^Model Context Protocol: Claude가 외부 도구 및 서비스에 연결할 수 있게 해주는 표준) 서버 사용법 |
| 주의사항 및 경고 | 알려진 함정 회피 | 편집하면 안 되는 파일, 특이사항, 환경 요구사항 |

## 위치 옵션

| 위치 | 범위 | 용도 |
|---|---|---|
| `./CLAUDE.md` | 프로젝트 (공유) | 팀 표준, 아키텍처, 빌드 명령어 |
| `./.claude/CLAUDE.md` | 프로젝트 (공유) | 위와 동일, 대체 위치 |
| `~/.claude/CLAUDE.md` | 개인 (모든 프로젝트) | 모든 프로젝트에 걸친 개인 선호사항 |
| `./CLAUDE.local.md` | 개인 (이 프로젝트) | 개인 sandbox URL, 선호하는 테스트 데이터. [Gitignored](^자동으로 .gitignore에 추가되어 로컬에만 유지되며 버전 관리에 커밋되지 않습니다) |
| [관리형 정책 경로](^IT/DevOps가 관리하는 시스템 수준 경로. macOS: /Library/Application Support/ClaudeCode/CLAUDE.md, Linux: /etc/claude-code/CLAUDE.md, Windows: C:\Program Files\ClaudeCode\CLAUDE.md) | 조직 전체 | 회사 보안 정책, 규정 준수 규칙 |

더 구체적인 위치가 더 넓은 범위보다 우선합니다. Claude는 하위 디렉토리의 `CLAUDE.md` 파일도 자동으로 탐지하여 해당 디렉토리에서 작업할 때 필요에 따라 로드합니다.

## 파일 임포트

`@path/to/file` 문법을 사용하여 추가 파일을 컨텍스트로 가져올 수 있습니다:

```markdown
See @README.md for project overview.
Git workflow: @docs/git-instructions.md
```

- 상대 경로는 임포트를 포함하는 파일 기준으로 해석됩니다
- 임포트는 최대 5단계까지 체이닝할 수 있습니다
- 프로젝트에서 처음 사용할 때 승인 대화상자가 표시됩니다
- README, package.json 또는 워크플로우 가이드를 내용 복사 없이 가져오는 데 유용합니다

## 효과적인 지침 작성법

- **구체적으로 작성**: [2칸 들여쓰기 사용](^Claude가 정확히 따를 수 있는 구체적이고 검증 가능한 지침) (O) / [코드를 올바르게 포매팅](^Claude가 일관되게 따르기에 너무 모호합니다. '올바르게'가 무엇을 의미하나요?) (X)
- **간결하게 작성**: 200줄 이하를 목표로 합니다. 파일이 길수록 [준수율이 떨어집니다](^Claude가 추적해야 할 지침이 많을수록 각 지침을 덜 신뢰성 있게 따릅니다)
- **구조를 활용**: 마크다운 헤더와 글머리 기호를 사용합니다. Claude는 독자처럼 구조를 스캔합니다
- **충돌 방지**: 두 지침이 모순되면 Claude는 임의로 하나를 선택합니다. 주기적으로 검토하세요
- **큰 파일은 분할**: [@ 임포트](^위에서 설명한 @path/to/file 문법으로 다른 파일을 CLAUDE.md 컨텍스트로 가져옵니다) 또는 `.claude/rules/` 파일을 사용하여 긴 지침을 분리합니다

## 팁

- CLAUDE.md는 [컨텍스트이지 강제가 아닙니다](^Claude는 이를 읽고 따르려고 하지만 엄격한 보장은 없습니다. 구체적이고 모호하지 않은 지침이 가장 좋은 결과를 얻습니다). 구체적인 지침이 모호한 것보다 더 신뢰성 있게 따라집니다
- `/init`은 기존 프로젝트에서도 작동합니다. 덮어쓰지 않고 개선 사항을 제안합니다
- 관련 없는 작업 사이에 `/clear`를 사용하여 축적된 컨텍스트를 초기화하면서 CLAUDE.md는 로드된 상태로 유지합니다
- CLAUDE.md는 [/compact에서도 유지됩니다](^/compact를 실행하면 Claude는 디스크에서 CLAUDE.md를 다시 읽고 새로 주입합니다. 대화에서만 제공된 지침은 손실되지만 CLAUDE.md는 유지됩니다). 디스크에서 다시 읽혀 새로 주입됩니다
- 비밀 키, API 키 또는 자격 증명을 절대 포함하지 마세요. 공개 문서로 취급하세요
- 단순하게 시작하세요. 가상의 필요가 아닌 실제 불편함을 기반으로 확장하세요
